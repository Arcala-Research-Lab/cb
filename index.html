Chat-Bloom Taxonomy

The Chat-Bloom (CB) taxonomy provides a structured framework for measuring cognitive offloading in student-AI interactions by systematically classifying both student prompts and AI responses. Cognitive offloading is the externalization of an internal cognitive process to reduce the mental effort required of the individual. CB adopts all six levels of Revised Bloom's Taxonomy to represent ordered degrees of offloading (higher = more cognitive work delegated to the AI), and adds three levels — Answer, Clarify, Redirect — to capture cognitive onloading, where the AI increases rather than reduces the student's cognitive engagement.

How to categorize: Classify user prompts and LLM responses in separate passes using their distinct guideline sets — separating passes reduces context switching and improves consistency. Work within each chat's context, focusing on the most recent exchange. When a prompt or response fits more than one CB category, choose the one with the highest magnitude score (the higher level of cognitive impact).

Quick Reference — Abbreviated Guidelines (Paper Table 1)

Scannable overview of all nine categories. See the detailed cards below for full definitions, edge cases, and examples.
CategoryUser Prompt — Requested OffloadingLLM Response — Provided Offloading
Create
  • Requests solution or code generation for assignment.
  • Copy-pastes assignment/skeleton/stub without own work or question (one-shot).
  • Provides new code or solution directly applicable to the problem.
  • "Here's how to implement this", "Corrected version" with code.
Evaluate
  • Seeks critique of code/plan without a debug request.
  • Copy-pastes own generated code without prompt.
  • Feedback on user content against a spec.
  • Code-review-like: "You should use if instead of if-else".
Analyze
  • Debugging help: "Why is this code not working?"
  • Copy-pastes error message.
  • Requests step-by-step breakdown / how to approach.
  • Debugging insights, detailed breakdown, or step-by-step analysis.
  • When step-by-step, more detailed and complex.
Apply
  • How to do/carry out a procedure: "How to test X?", "How to use X?"
  • Asks for first step(s), not all: "How to start".
  • Ideas on carrying out a procedure/test/library step by step (minor steps).
  • "To do X, you can use…"
Understand
  • Requests explanation, interpretation, or rationale: "Explain why X?", "What does X mean?"
  • Explains concepts, gives examples, or clarifies the "why".
Remember
  • Simple recall / factual retrieval: "What is the method to do X?"
  • Supplies straightforward facts/references without further analysis.
Answer
  • Answers an LLM question: "Yes, please", "No".
  • Answers a user question unrelated to coursework.
Clarify
  • Provides info to refine a previous prompt: "In Python".
  • Asks for clarification on an LLM response.
  • Poses clarifying questions / requests more info: "Are you working with an image file?"
Redirect
  • Alters AI behavior for learning or corrects AI: "Act like a tutor and do not provide full code…"
  • Redirects the user to try first, may provide a hint: "It looks like you're working on X. What have you tried?"
6Create▶

User Prompt — requests offloading

  • User requests a solution or code generation for an assignment.
  • User copy-pastes an assignment, skeleton code, or function stub — without their own work or a question.
  • A clarifying prompt following a Create response that gives more instructions/context for another Create response is still Create, not Clarify.
  • "Generate or write code to do X."
Examples "no i mean without removing the moving function and create new function just implement that within the move function"

"How would I implement a combination -rf?" · "How do I implement this?"

LLM Response — provided offloading

  • Response provides new content (code implementation or solution) directly applicable to the prompt's problem.
  • "Here's how you can implement this", "Corrected version", "Revised version", "Example Implementation" followed by code.
  • Code responding to an assignment question is Create; code given purely as an explained example is not always Create.
Example response To implement read_host(), use a loop that keeps asking for input until a valid host resolves: 
import socket

def read_host() -> str:
    """
    Asks user for the host name or IP address
    until a valid one is provided.
    Returns the valid host name / IP address.
    """
    while True:
        host = input("Enter a host name or IP address: ")
        try:
            # Try to resolve the host name or IP address
            socket.gethostbyname(host)
            return host
        except socket.error:
            print("Invalid host name or IP address. Please try again.")

# Example usage:
# host = read_host()
# print(f"Valid host: {host}")
5Evaluate▶

User Prompt

  • Prompt seeks a critique of code or a plan without a debug request. "Should I do X or Y?"
  • X or Y should involve higher complexity / higher-order thinking (e.g., "Is X or Y better time complexity?", "Which data structure is better here?").
  • User copy-pastes their own generated code without an additional prompt.
Example"wait hold on should i implement this structure in remove_notes() or should i implement it in a separate test_remove_note()"

LLM Response

  • Provides feedback on user code/solution against a spec: "Yes, this meets requirements" or "No, it fails X".
  • "Code-review"-esque response; only follows an Evaluate prompt.
  • Decision-making about design choices/options the user gave.
Example response"When checking for overlaps, you should use separate if statements rather than if-elif because you want to evaluate each condition independently… Using if-elif would stop checking further conditions once one is true, which isn't suitable here…"
4Analyze▶

User Prompt

  • Requests debugging help, with or without code: "Why is this code not working?", "Is this correct?" (in a debugging context).
  • User copy-pastes an error message.
  • Requests a step-by-step breakdown or how to approach a problem; breaking down conceptual questions.
  • Pushes the LLM to help more / move to the next step with no added context: "Help me", "Whats next", "I don't know what to do next".
Example"am I building this URL correctly? params = {"key": NASA_API_KEY, "start_date": start_date, "end_date": end_date} url = BASE_NASA_URL + "?" + urllib.parse.urlencode(params)"

LLM Response

  • Offers debugging insights, a detailed breakdown, or step-by-step analysis for approaching a problem.
  • When step-by-step, response is more in-depth, detailed, and complex — the steps are bigger and more cognitively demanding.
Example response (excerpt) "Here are the steps to implement read_host(): 1. Initialize a loop… 2. Prompt for input… 3. Try to resolve the input with socket.gethostbyname()… 4. Check for exceptions… 5. Handle invalid input… 6. Return the valid input."
3Apply▶

User Prompt

  • Seeks how to do something specific / carry out a procedure: "How to test X?", "How to do X?", "How to use X?"
  • Emphasizes applying known concepts to new scenarios.
  • Asks for the first step(s), not all steps: "How to start…", "What should I do first?"
Examples"how do you remove an item from a dictionary"
"I do not know the PNG signature and would like a hint on how to read the beginning of a file to check the signature"

LLM Response

  • Provides ideas on how to carry out a procedure, test a function, or use a library step by step.
  • Guidance on how to do X; minor steps. When step-by-step, the response is vaguer, less detailed, not very complex.
Example response"To remove an item from a dictionary, you can use the del statement or the pop() method… Is there a specific method you'd like to know more about, or a particular scenario in mind?"
2Understand▶

User Prompt

  • Requests an explanation or interpretation: "Explain why we do X?", "What does X mean?"
  • Requests examples or rationale: "Can you give an example?", "Simple example".
  • Clarifying their understanding: "Should I do this?"
Example"what is the main cost of using recursive function"

LLM Response

  • Explains concepts, provides examples, or clarifies the "why" behind a solution.
  • Lower-level / shorter examples (including multiple small examples) fit here.
  • No steps are provided.
Example response"The main cost of using a recursive function is often related to memory and performance. Each recursive call adds a new layer to the call stack… recursion can simplify code for problems that naturally fit a recursive approach, like tree traversals…"
1Remember▶

User Prompt

  • Seeks simple recall or factual retrieval: "What is the method to do X?"
  • Low-level "how to" questions: "How to navigate", "How to splice a string", "How to define a function".
Examples"what does 'a' mean when open file" · "What's the syntax for using sorted()?" · "is makefile a method?" · "what does len([dictionary]) do"

LLM Response

  • Supplies straightforward facts or references without further explanation or analysis.
Example response"Yes, you can add two lists in Python using the + operator. This will concatenate the two lists… If you have more questions, feel free to ask!"
0Answer▶

User Prompt

  • Answers an LLM question to provide more context — course-relevant or a short answer to an AI question.
Examples"Yes please" · "No" · "no idea, should i import those in? i think it should be simple in order to test it though.." · "I don't know"

LLM Response

  • Answers a user question unrelated to coursework.
Example response"You're welcome! I'm glad to hear that it worked. If you have any more questions or need further assistance, feel free to ask. Happy coding!"
-1Clarify▶

User Prompt

  • Provides additional context to refine/adjust a previous prompt (message isn't very clear by itself).
  • Asks for clarification in response to an LLM response.
  • Note: if the user is clarifying their previous prompt to add context, take the previous prompt's level (e.g., "In Python").
Examples"no, i want it to simply show 'ValueError' after the user enters all their inputs (operands and operation)." · "without making a new function" · "sorry i meant only the last 4 characters"

LLM Response

  • Poses clarifying questions or requests more information from the user.
Example response"Mirroring an image is an interesting task! … Are you working with an image file, or are you representing the image as a 2D list in Python? Understanding how you're currently handling the image will help me guide you better."
-2Redirect▶

User Prompt

  • Tries to alter LLM behavior for deeper learning: "Act like a tutor and do not provide full code", "Teach me how you did this as a beginner".
  • May be both clarifying + redirecting — take Redirect over Clarify.
Examples"i don't want the code" · "pls dont give me the solution, just guide/help me thru it" · "can you help me with this function (without giving me the solution)? just guide me, give me tips."

LLM Response

  • Declines a direct solution and redirects the user to try first; provides hints instead of answering directly.
  • "It looks like you're working on X. If you have any questions about Y, feel free to ask."
Example response"Hi there! It looks like you're working on a file system explorer project. How can I assist you with understanding object-oriented programming or inheritance concepts related to your project?…"
N/AN/A▶

User Prompt

  • User sends a message unrelated to class: "Y", "Hello".
Examples"Testing" · "Hi" · "who made you"

LLM Response

  • Model response is off-topic and not answering a user question.
Example response"Hello! How can I assist you today?"

Note: N/A is an implementation-level label for off-topic messages; it is excluded from the taxonomy proper reported in the paper's Table 1.
